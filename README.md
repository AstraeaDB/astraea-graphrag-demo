# GraphRAG Demo: A Tale of Two Cities

A demonstration of how a modest 4B-parameter LLM can answer complex literary questions with minimal hallucination by using structured knowledge graph access. The system combines **AstraeaDB** (a vector-property graph database), **Ollama** (local LLM inference with gemma3:4b), and **MCP** (Model Context Protocol) to show that graph structure compensates for model size.

## How It Works

The demo builds a knowledge graph from Charles Dickens's *A Tale of Two Cities* containing 229 nodes (characters, locations, events, themes, chapters, and text passages) connected by 317 edges (relationships, locations, event participation, thematic associations, and provenance links to source text). Every node carries a 128-dimensional embedding for semantic search.

When you ask a question, the system:

1. Embeds your question using EmbeddingGemma (Matryoshka truncation to 128 dims)
2. Sends the question to gemma3:4b with tool descriptions in the system prompt
3. The model outputs structured tool calls (`TOOL_CALL: {"tool": "...", "args": {...}}`), which the orchestrator parses and executes against AstraeaDB's MCP server via JSON-RPC over stdio
4. Tool results (enriched with node names and edge types) are fed back to the model, which can make further tool calls (up to 5 rounds)
5. The model generates a final answer grounded in the graph data

An optional comparison mode shows the same question answered by the raw LLM (no graph) alongside the GraphRAG-augmented answer, making the improvement visible.

**Note on tool calling:** gemma3:4b does not support Ollama's native tool-use API, so the orchestrator implements tool calling through prompt engineering. The model is instructed to output tool calls in a structured JSON format, which the orchestrator parses, executes via MCP, and feeds back.

### Embeddings

All 229 nodes have 128-dimensional embeddings generated by EmbeddingGemma via Ollama. The model produces 768-dimensional vectors; we use Matryoshka representation learning to truncate to 128 dimensions and L2 re-normalize. This matches AstraeaDB's HNSW vector index configuration and enables semantic search alongside graph traversal.

## How Semantic Embeddings and HNSW Work Together

### The Embedding Pipeline

#### 1. Generating the vectors (offline, at build time)

Every node in the knowledge graph -- characters, locations, events, themes, chapters, and text chunks -- gets a **128-dimensional embedding vector** derived from its textual content.

The process (`scripts/generate_embeddings.py`):

1. **Text extraction** -- Each entity type contributes different text. A character gets `"Sydney Carton (also known as: Carton, the jackal). A dissolute English barrister who..."`, a text chunk gets its raw prose, a chapter gets `"title. summary"`, etc.

2. **EmbeddingGemma via Ollama** -- The text is sent to Ollama's `/api/embed` endpoint using the `embeddinggemma` model, which returns **768-dimensional** vectors.

3. **Matryoshka truncation** -- The 768-dim vector is sliced to just the **first 128 dimensions** (`emb[:128]`). This works because EmbeddingGemma is trained with **Matryoshka Representation Learning** -- a technique where the model is trained so that *any prefix* of the embedding is a valid, useful embedding at that dimensionality. The first 128 dims aren't random garbage; they're trained to capture the most important semantic signal even at that reduced size.

4. **L2 re-normalization** -- After truncation, the vector is no longer unit-length, so it's re-normalized: divide each component by the vector's L2 norm. This is critical because cosine similarity (which HNSW uses for search) assumes unit vectors.

The result: `data/embeddings.json`, a map from every entity string ID to a 128-dim float vector.

#### 2. Loading vectors into AstraeaDB's HNSW index (offline, at build time)

In `scripts/load_graph.py`, every `create_node` call passes `embedding=emb`:

```python
node_id = client.create_node([label], props, embedding=emb)
```

When AstraeaDB receives a node with an embedding, it inserts that vector into its **HNSW (Hierarchical Navigable Small World) index** -- a data structure configured for dimension=128 and cosine distance.

#### 3. What HNSW actually does

HNSW is a graph-based approximate nearest neighbor index. Think of it as a **skip-list built on a proximity graph**:

- **Layer 0 (bottom)**: Every embedded node lives here, connected to its nearest neighbors in vector space. So "Sydney Carton" is linked to "Charles Darnay" (they're semantically similar -- both English protagonists), not because of any knowledge graph edge, but because their description embeddings are close in 128-dim space.

- **Higher layers**: Progressively sparser random subsets of nodes, each connected to their nearest neighbors at that layer. These form "express lanes" for search.

- **Search**: To find the k nearest neighbors to a query vector, HNSW enters at the top layer, greedily hops toward the closest node, drops down a layer, and repeats. By the time it reaches layer 0, it's in the right neighborhood and does a fine-grained local search. This gives **O(log n)** search time instead of brute-force O(n).

The key insight: **HNSW is a separate graph from the knowledge graph.** AstraeaDB maintains two graph structures simultaneously:
- The **knowledge graph** (characters -> events -> locations, with typed edges like `RELATED_TO`, `PARTICIPATES_IN`)
- The **HNSW vector index** (a navigational structure over the same nodes, but edges represent proximity in embedding space)

#### 4. How they're used together at query time

When a user asks a question, the orchestrator (`src/orchestrator.py`) does this:

1. **Embeds the question** using the same pipeline -- EmbeddingGemma -> truncate to 128-dim -> L2-normalize (`src/embeddings.py`). This produces a query vector in the same semantic space as all the node embeddings.

2. **The model calls `vector_search`** -- The MCP bridge sends the query vector to AstraeaDB, which runs HNSW search to find the k closest node embeddings. This answers: "which entities in the graph are semantically most related to this question?" without knowing anything about graph structure.

3. **The model calls `hybrid_search`** -- This is where the two systems truly combine. `hybrid_search` takes both an **anchor node** (a graph position) and a **query vector**, then blends:
   - **Graph proximity** (BFS distance from the anchor, up to `max_hops`)
   - **Vector similarity** (cosine distance from the query embedding via HNSW)

   The `alpha` parameter controls the blend: 0.0 = pure graph distance, 1.0 = pure vector similarity, 0.5 = equal weight. This finds nodes that are *both* topologically close in the knowledge graph *and* semantically relevant to the question.

### Concrete Example

Question: *"What is Sydney Carton's relationship with Charles Darnay?"*

1. The question is embedded -> 128-dim vector capturing the semantics of "Carton + Darnay + relationship"

2. **`vector_search`** with that vector finds nodes whose descriptions are semantically closest: probably the Carton node, the Darnay node, text chunks about their trial scene, the "duality" theme node -- all without any graph traversal.

3. The model then calls **`find_by_label("Character")`** -> gets Carton's node ID -> calls **`hybrid_search(anchor=carton_id, query=question_embedding)`** -> finds entities that are both *near Carton in the graph* (his neighbors, his events, his locations) AND *semantically related to the question*. This surfaces the trial scene, the resemblance, Lucie's marriage -- graph-connected context filtered by semantic relevance.

4. The model can also call **`neighbors`** and **`shortest_path`** for pure graph traversal (no vectors involved) to find the explicit `RESEMBLES` edge between Carton and Darnay.

### Component Summary

| Component | Role |
|-----------|------|
| **EmbeddingGemma + Matryoshka** | Converts text -> 128-dim semantic vectors (same space for nodes and queries) |
| **HNSW index** | Fast approximate nearest-neighbor lookup over those vectors (O(log n) instead of O(n)) |
| **Knowledge graph** | Typed relationships between entities (family ties, event participation, thematic links) |
| **`vector_search`** | Pure HNSW -- "what nodes mean something similar to this question?" |
| **`hybrid_search`** | HNSW + BFS blended -- "what's both graph-close and semantically relevant?" |

The embeddings give the system **semantic understanding** (what things *mean*), HNSW makes that understanding **searchable at speed**, and the knowledge graph gives it **structural reasoning** (how things *relate*). The three together let a 4B-parameter model answer complex multi-hop literary questions it couldn't handle from its training data alone.

## Prerequisites

- **Ollama** with two models:
  ```bash
  ollama pull gemma3:4b
  ollama pull embeddinggemma
  ```
- **AstraeaDB** built from source (the debug build includes the `mcp` subcommand)
  ```bash
  cd /path/to/astraeadb
  cargo build -p astraea-cli
  ```
- **Python 3.9+** with requests:
  ```bash
  pip install -r requirements.txt
  ```

## Available MCP Tools

The LLM has access to 10 tools from AstraeaDB's MCP server. The MCP bridge enriches results from `find_by_label`, `neighbors`, and `vector_search` with human-readable node names, labels, and edge types so the model can navigate the graph effectively.

| Tool | Purpose |
|------|---------|
| `find_by_label` | Find all nodes of a type (Character, Location, Event, etc.) -- enriched with names |
| `get_node` | Get full details of a node by ID |
| `neighbors` | Explore a node's relationships -- enriched with edge types and target names |
| `shortest_path` | Find the connection path between two nodes |
| `vector_search` | Semantic similarity search using embeddings -- enriched with names and descriptions |
| `hybrid_search` | Combined vector + graph proximity search |
| `extract_subgraph` | Get a readable text summary of a node's neighborhood |
| `bfs` | Breadth-first traversal from a node |
| `query` | Execute GQL (Graph Query Language) queries |
| `graph_stats` | Knowledge graph statistics |

## How the Knowledge Graph Was Built

1. The full text of *A Tale of Two Cities* (Project Gutenberg) was processed book-by-book
2. Claude extracted characters, locations, events, chapter summaries, and key text passages with structured JSON output
3. Coreference resolution merged duplicate entities (e.g., "Dr. Manette" / "Alexandre Manette" / "One Hundred and Five, North Tower")
4. Themes and thematic edges (EMBODIES) were added based on literary analysis
5. Per-book extractions were merged via `scripts/merge_extractions.py`, normalizing IDs and deduplicating
6. All entity descriptions and text passages were embedded using EmbeddingGemma (768-dim, truncated to 128 via Matryoshka)
7. Everything was loaded into AstraeaDB with embeddings attached to nodes for HNSW indexing
